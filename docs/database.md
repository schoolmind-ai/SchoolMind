# Database Architecture

## Runtime Engines

SchoolMind now has explicit database engine selection:

- `DATABASE_ENGINE=sqlite`: local development and automated tests.
- `DATABASE_ENGINE=postgres`: production runtime using Supabase managed PostgreSQL.

Production refuses SQLite unless `APP_ENV=demo` or `ALLOW_SQLITE_IN_PRODUCTION=true` is explicitly set. That exception is for controlled demos only.

## Adapter Layer

The app keeps its lightweight raw-SQL style but routes database access through `schoolmind/db.py`:

- request-scoped connection reuse through Flask `g`
- `query_one`, `query_all`, `execute`, `executemany`
- transaction context helper
- placeholder conversion from SQLite `?` to PostgreSQL `%s`
- dict-like row access for both `sqlite3.Row` and `psycopg.rows.dict_row`
- connection close on app-context teardown

The PostgreSQL driver is `psycopg[binary]`.

`schema_migrations` stores applied schema versions. Schema initialization is idempotent and create-if-not-exists.

## PostgreSQL Compatibility

The PostgreSQL schema generator removes SQLite-only syntax:

- no `PRAGMA`
- no `AUTOINCREMENT`
- PostgreSQL identity columns use `GENERATED BY DEFAULT AS IDENTITY`
- existing text date fields remain text-compatible for safe migration
- indexes cover user lookup, role/school filters, billing, coupons, tokens, AI/Nour messages, games, support workflows, audit logs, and security events

## Backup Model

The app-level workspace JSON export remains available and includes preferences, personalization flags, Nour messages, games activity, billing, coupons, activity/audit, consent, subscriptions, and support workflow tables.

For production, Supabase project backups are the database-level backup source. Use app JSON export for tenant-level operational export, not as the only disaster-recovery mechanism.
